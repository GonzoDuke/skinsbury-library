# Carnegie — v1.1 Feature Plan

**Prerequisite:** Read `STATUS-V1.0.md` for full architectural context before starting any work.

This document specifies six features in priority order. Implement them in this order — each one is a standalone commit that can be tested independently. Do not bundle.

---

## Feature 1: "Add missing book" on Review

**Priority:** Highest. This is the most common user frustration — Pass A misses a spine and there's no way to recover without re-photographing.

**What it does:** On the Review screen, the user can click a button to add a book that Pass A failed to detect. Two paths:

### Path A: Draw on the source photo
- Add an "Add missing book" button at the bottom of each batch group on the Review screen.
- Clicking it opens a modal showing the full source photo for that batch at high resolution.
- The user draws a rectangle over the missed spine (click and drag). Show the rectangle as a dashed orange (#C85A12) border while drawing.
- On release, the app:
  1. Crops the selected region from the full-resolution source image (same as `cropSpine()` does for Pass A detections)
  2. Sends the crop to `/api/read-spine` (Pass B, Opus)
  3. Runs the standard pipeline: lookup → ground → infer-tags
  4. Inserts the resulting BookRecord into the batch, marked with a "Manually added" badge
  5. The card appears in the review list with all the normal editing/approve/reject controls

### Path B: Manual entry
- Below the "draw on photo" option in the same modal, include a "Or enter manually" section.
- Three fields: Title, Author, ISBN (optional).
- Submitting skips Pass B entirely, runs lookup → infer-tags using the typed values.
- The resulting BookRecord gets a "Manual entry" badge instead of a spine thumbnail.

**Files to change:**
- `app/review/page.tsx` — add the "Add missing book" button per batch group
- New component: `components/SpineSelector.tsx` — the modal with the photo canvas and draw-to-select interaction
- `lib/pipeline.ts` — new function `addManualBook(title, author, isbn?, batchId)` that runs lookup + tags
- `lib/store.tsx` — new action `ADD_BOOK` to insert into the correct batch
- `lib/types.ts` — add `manuallyAdded?: boolean` to BookRecord

**Test:** Open a batch where Pass A missed a book. Use Path A to draw a box around it. Confirm the book appears in the review list with correct metadata. Then use Path B to type a title/author for a book not in any photo. Confirm it appears with a "Manual entry" badge and correct tags.

## Feature 2: OCLC Classify as t5 lookup tier

**Priority:** High. Fills remaining LCC gaps when Open Library and LoC SRU both miss. Free, no key required, no institutional subscription.

**What it does:** Adds a fifth tier to the lookup cascade in `lib/book-lookup.ts` that queries OCLC's Classify API specifically for LCC and DDC classification numbers. This tier fires only when LCC is still missing after tiers 1–4.

### API details
- **Endpoint:** `http://classify.oclc.org/classify2/Classify`
- **Auth:** None. Free and unrestricted.
- **No API key needed. No env var needed.**
- **Query by ISBN (preferred):**
  `http://classify.oclc.org/classify2/Classify?isbn={isbn}&summary=true`
- **Query by title + author (fallback):**
  `http://classify.oclc.org/classify2/Classify?title={title}&author={author}&summary=true`
- **Response format:** XML. Parse it for:
  - `<lcc>` element → `sfa` attribute contains the most frequently assigned LCC number across all OCLC member libraries
  - `<ddc>` element → `sfa` attribute contains the most frequently assigned DDC number
  - `<work>` element → `title` and `author` attributes for verification
- **Response codes in the XML:**
  - `response code="0"` — no results
  - `response code="2"` — single work found (ideal)
  - `response code="4"` — multiple works found; use the `<works>` list and pick the best match by title/author similarity
  - `response code="102"` — unexpected error

### Integration into the cascade
- In `lookupBook()`, after all existing tiers (ol-t1 through gb) and after the LoC SRU enrichment step, check if `lcc` is still empty/undefined.
- If LCC is missing AND we have an ISBN: query Classify by ISBN.
- If LCC is missing AND we don't have an ISBN: query Classify by title + author.
- If Classify returns an LCC, add it to the BookLookupResult. Do NOT overwrite an existing LCC — Classify is a fallback, not an authority override.
- Also grab DDC if returned — store it on the BookRecord as `ddc?: string` for potential future use (LibraryThing displays DDC).
- Set `tier: 'oclc-classify'` in the result for diagnostics.

### Important: this tier does NOT provide ISBN, publisher, year, or subjects
- Classify only returns classification numbers (LCC, DDC). It does not replace any other tier for bibliographic metadata.
- It's purely an LCC gap-filler. If the book already has LCC from any earlier tier, skip this call entirely.

### In the BookCard
- When LCC comes from OCLC Classify, show a small "from OCLC" badge next to the LCC field (similar to the existing "from spine" badge). Use a neutral gray — this is less authoritative than a spine read or LoC SRU, but better than nothing.
- The badge hierarchy for LCC provenance: "from spine" (most authoritative) > "from LoC" > "from OCLC" > "from OL" (least authoritative, no badge)

**Files to change:**
- `lib/book-lookup.ts` — add `lookupOclcClassify(isbn?, title?, author?)` function, integrate after LoC SRU step
- `lib/types.ts` — add `ddc?: string` to BookRecord, add `'oclc-classify'` to tier union, add `lccSource?: 'spine' | 'loc' | 'oclc' | 'ol'` for provenance tracking
- `components/BookCard.tsx` — add "from OCLC" badge, implement badge hierarchy

**Test:** Find a book where the current 4-tier chain returns everything except LCC (check the dev console for a case where LoC SRU also missed). Confirm that OCLC Classify fills in the LCC. Confirm the "from OCLC" badge appears. Confirm that if a book already has LCC from an earlier tier, Classify is not called.

## Feature 3: Approved-tag feedback loop

**Priority:** Medium. Small effort, high long-term value. Every approved batch makes the next batch smarter.

**What it does:** When the user exports a batch, any `[Proposed]` tags on approved books are automatically promoted into the vocabulary and logged.

### Implementation

This runs at export time, not at approve time. When the user clicks "Download CSV" on the Export screen:

1. Scan all books in the export set for tags prefixed with `[Proposed]`.
2. For each proposed tag:
   a. Strip the `[Proposed]` prefix.
   b. Determine which domain it belongs to:
      - Look at the book's LCC code and match against domain LCC prefixes.
      - If no LCC match, look at the other tags on the same book and use the domain of the majority.
      - If still ambiguous, place in `_unclassified`.
   c. Add the tag to the appropriate domain's tag array in `tag-vocabulary.json`.
   d. Append a line to `vocabulary-changelog.md`: `- {date}: Added "{tag}" to {domain} (from batch "{batchLabel}", book "{title}")`.
3. Write the updated `tag-vocabulary.json` to the repo.

### Problem: Vercel can't write to the repo at runtime

This is a serverless app — we can't write files to the filesystem in production. Two options:

**Option A (recommended for v1.1):** Do it client-side. When the export triggers, the app identifies proposed tags, builds the updated vocabulary JSON and changelog entry, and offers them as a second download: "Download updated vocabulary (2 new tags)". The user drops those files into their repo and pushes. Low-tech but it works.

**Option B (future):** Create a `/api/update-vocabulary` route that commits directly to the GitHub repo via the GitHub API. Requires a `GITHUB_TOKEN` env var with repo write access. Elegant but more setup.

Implement Option A for now.

**UI changes:**
- On the Export screen, below the CSV download button, show a section: "Vocabulary updates" listing any proposed tags being promoted, with their assigned domains. Include a "Download updated vocabulary files" button that downloads a zip containing the new `tag-vocabulary.json` and an appendable `vocabulary-changelog-additions.md`.
- In the CSV itself, strip the `[Proposed]` prefix from all tags — the export should contain clean tags.

**Files to change:**
- `app/export/page.tsx` — add vocabulary update section
- `lib/csv-export.ts` — strip `[Proposed]` prefix from tags during export
- New: `lib/vocabulary-update.ts` — functions to identify proposed tags, assign domains, generate updated JSON and changelog

**Test:** Process a photo that produces a `[Proposed]` tag. Approve the book. Go to Export. Confirm the proposed tag appears in the vocabulary updates section with a domain assignment. Download the vocabulary files. Confirm the JSON is valid and contains the new tag in the correct domain.

## Feature 4: Bulk re-tag

**Priority:** Medium. Becomes important as the vocabulary grows and the user wants consistency across previously cataloged books.

**What it does:** A button on the Review screen that re-runs tag inference on selected books without touching any other metadata. Used when the vocabulary has been updated (e.g., new "Drama" and "Shakespeare" tags added) and the user wants to propagate those to books that were tagged before those categories existed.

### UI

- Add a "Bulk actions" dropdown to the Review screen header, next to the existing filter/sort controls.
- Options:
  1. **Re-tag all approved** — runs tag inference on every approved book in the current view (respecting active filters).
  2. **Re-tag by domain** — opens a dropdown showing all 12 domains. Selecting one re-tags only books whose LCC falls in that domain.
  3. **Re-tag selected** — if we add checkboxes to BookCards (see below), re-tag only checked books.
- Before running, show a confirmation: "This will replace existing tags on N books with fresh inferences from the current vocabulary. User-added tags will be preserved. Continue?"

### Behavior

- For each targeted book, call `/api/infer-tags` with the book's current metadata (title, author, LCC, subjects, etc.) and the CURRENT vocabulary (which may have new tags since the book was first processed).
- The response replaces the book's `genreTags` and `formTags` UNLESS the user has manually added or removed tags since the last inference. To detect this: compare current tags to `book.original.genreTags` — if they differ, the user has edited, so MERGE instead of replace (keep user additions, add new inferred tags, don't remove user-added tags).
- After re-tagging, each affected BookCard briefly flashes to indicate it was updated, and the reasoning field updates with new reasoning.
- Re-tagged books retain their approved/pending/rejected status.

### Add checkboxes to BookCards

- Add a checkbox to the left side of each BookCard.
- When any box is checked, show a floating action bar at the bottom: "N selected · Re-tag · Approve all · Reject all"
- This is useful beyond bulk re-tag — bulk approve and bulk reject are natural companions.

**Files to change:**
- `app/review/page.tsx` — add bulk actions dropdown, floating action bar, checkbox state
- `components/BookCard.tsx` — add checkbox
- `lib/store.tsx` — new action `BULK_RETAG` that takes an array of book IDs, calls infer-tags for each, and merges results
- `lib/pipeline.ts` — new function `retagBook(book: BookRecord)` that calls `/api/infer-tags` and handles the merge logic

**Test:** Add "Drama" and "Shakespeare" to the vocabulary. Process a shelf that includes Shakespeare. Approve the books. Now use "Re-tag by domain: Literature." Confirm that Shakespeare books gain the new tags without losing any user-edited tags. Confirm books outside the Literature domain are untouched.

## Feature 5: Per-spine model selector

**Priority:** Low-medium. Cost optimization. Only matters if API costs become a concern.

**What it does:** Uses Sonnet (cheaper) for spines that are visually easy to read, and Opus (expensive, more accurate) only for the hard ones. Reduces Pass B cost by ~50% with no accuracy regression on clear spines.

### Selection logic

After Pass A returns bounding boxes, classify each spine before sending to Pass B:

```
spineArea = detection.width * detection.height  (as % of image area)
aspectRatio = detection.height / detection.width

if spineArea > 2.0% AND aspectRatio < 3:
  → EASY (likely wide horizontal spine, large text)
  → Use Sonnet

else:
  → HARD (narrow vertical spine, small text)
  → Use Opus
```

The thresholds above are starting points. They should be tunable — store them in a config object at the top of `lib/pipeline.ts`:

```typescript
const SPINE_MODEL_CONFIG = {
  easyAreaThreshold: 2.0,     // % of image area
  easyAspectMaxRatio: 3,      // height/width
  easyModel: 'claude-sonnet-4-20250514',
  hardModel: 'claude-opus-4-7',
};
```

### API route changes

- `/api/read-spine` currently hardcodes Opus. Change it to accept an optional `model` parameter in the request body. Default to Opus if not specified (backward compatible).
- `lib/pipeline.ts` — in `buildBookFromCrop()`, compute the spine classification and pass the model choice to the read-spine call.
- BookCard: add a subtle indicator showing which model read the spine (e.g., a tiny "S" or "O" next to the confidence badge). This helps with debugging if Sonnet produces a bad read — the user can reread with Opus.

### Fallback

- If a Sonnet read returns confidence LOW, automatically retry with Opus before presenting to the user. This catches cases where the area heuristic misjudged difficulty.
- Log both attempts in the dev console for tuning the thresholds.

**Files to change:**
- `app/api/read-spine/route.ts` — accept `model` param, default to Opus
- `lib/pipeline.ts` — spine classification logic, pass model to read-spine, auto-retry on LOW
- `components/BookCard.tsx` — model indicator
- `lib/types.ts` — add `ocrModel?: string` to BookRecord

**Test:** Process a photo with a mix of wide (easy) and narrow (hard) spines. Confirm Sonnet is used for the wide ones and Opus for the narrow ones (check dev console logs). Confirm that if Sonnet returns LOW confidence, it auto-retries with Opus. Confirm no accuracy regression compared to all-Opus.

## Feature 6: PWA / Add to Home Screen

**Priority:** Lowest. Nice-to-have for phone-based cataloging sessions.

**What it does:** Makes the app installable as a Progressive Web App so the user can tap an icon on their phone's home screen and go straight to the upload screen with the camera.

### Implementation

1. **Web App Manifest** — create `public/manifest.json`:
```json
{
  "name": "Carnegie",
  "short_name": "Carnegie",
  "start_url": "/",
  "display": "standalone",
  "background_color": "#FAFAF7",
  "theme_color": "#C85A12",
  "icons": [
    { "src": "/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "/icon-512.png", "sizes": "512x512", "type": "image/png" }
  ]
}
```

2. **Icons** — generate a simple app icon. Use a book + camera motif, or just the initials "TLS" in Source Serif on an orange (#C85A12) background. Create at 192x192 and 512x512.

3. **Link the manifest** in `app/layout.tsx`:
```html
<link rel="manifest" href="/manifest.json" />
<meta name="theme-color" content="#C85A12" />
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta name="apple-mobile-web-app-status-bar-style" content="default" />
```

4. **Service worker** — for basic PWA installability, a minimal service worker is required. Create `public/sw.js`:
```javascript
self.addEventListener('fetch', () => {});
```
Register it in `app/layout.tsx`:
```javascript
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js');
}
```
We don't need offline support — the app requires API calls to function. The service worker just enables the "Add to Home Screen" prompt.

5. **Camera input on mobile** — update `PhotoUploader.tsx` to detect mobile and show a "Take photo" button that opens the camera directly:
```html
<input type="file" accept="image/*" capture="environment" />
```
The `capture="environment"` attribute opens the rear camera. This sits alongside the existing file picker for gallery uploads.

**Files to change:**
- `public/manifest.json` (new)
- `public/icon-192.png`, `public/icon-512.png` (new)
- `public/sw.js` (new)
- `app/layout.tsx` — manifest link, service worker registration
- `components/PhotoUploader.tsx` — mobile camera input

**Test:** Deploy to Vercel. Open on a phone. Confirm the browser offers "Add to Home Screen." Install. Open from home screen — confirm it launches in standalone mode (no browser chrome). Confirm the camera button opens the rear camera. Take a photo and confirm it enters the upload queue at full resolution.
